レストラン予約APIをRESTで設計する

〜パブリックとBearer、/me パスとステータス〜

シナリオの前提

  • 目的： 飲食店の予約サイトのバックエンドAPI
  • 空き状況の閲覧 → ゲスト（未ログイン）でも可
  • 予約の新規 / 自分の一覧 / キャンセル → 認証（Bearer）必須

リソースの地図（典型パス）

名詞の複数形＋ /{id} で1件。空きは availability を1リソースに切る例。

① 空き状況 GET /availability

  • 200 OK · レスポンス本文 json
  • public → Authorization なし
  • 読み取り： Accept: application/json を付与しがち

② 新規予約 POST /reservations

  • 201 Created · 予約IDなどを json で返す想定
  • bearer Authorization + Content-Type: application/json

③ 自分の予約一覧 GET /me/reservations

「どのユーザーか」をURLに生のIDで載せず、トークン＝私に閉じる典型。

/users/12345/reservations 他者のIDの列挙が露見しやすい （管理画面向けの別文脈では有用）
/me/reservations Bearer とセットで「自分の束」

別解: /users/me/reservations ・ GET /reservations（文脈で“自分”と決め打ち）

成功 200 ・ Accept + Authorization

④ キャンセル DELETE /reservations/{id}

  • 成功の定番：204 No Content（レスポンス本文なし）
  • Authorization のみ（送るJSONが無いので通常 Content-Type 不要）
  • 別解: 200 + キャンセル結果JSON（チーム方針次第）

4本のエンドポイント一覧

操作MethodPathAuth主な status

予約作成 POST /reservations bearer 201
自分の一覧 GET /me/reservations 他に別解あり bearer 200
キャンセル DELETE /reservations/{id} bearer 204 (200+JSON 別解)
