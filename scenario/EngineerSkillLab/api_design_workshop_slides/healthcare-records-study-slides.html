ヘルスケア記録APIをRESTで設計する

個人データを扱う前提で、全エンドポイント bearer の5問を整理

シナリオ前提

  • ウェアラブル連携の健康アプリ（個人のバイタル・同意・共有設定）
  • この課題では 全エンドポイントが bearer 認証必須
  • 取得系は Accept、書き込み系は Content-Type を基本に使い分ける

エンドポイント全体図（5本）

一覧取得：GET /records

  • 用途：自分の健康記録一覧を取得する
  • 認証：bearer
  • 成功：200 OK + JSON
  • ヘッダ：Accept + Authorization
  • 別解：GET /me/records

記録作成：POST /vitals

  • 用途：心拍・歩数などのバイタルを新規記録する
  • 認証：bearer
  • 成功：201 Created + JSON
  • ヘッダ：Content-Type + Authorization
  • 別解：POST /records/vitals

同意更新：PUT /consents/{id}

  • 用途：医療機関へのデータ共有同意を更新する
  • 認証：bearer
  • 成功：200 OK + JSON
  • 更新系なので JSON 本文を持つ（Content-Type 必須）
  • 別解：PATCH /consents/{id}

共有先一覧：GET /providers

  • 用途：共有先の医療機関一覧を取得する
  • 認証：bearer
  • 成功：200 OK + JSON
  • ヘッダ：Accept + Authorization
  • 別解：GET /sharing/providers

共有取消：DELETE /sharing/{id}

  • 用途：特定の共有設定を取り消す
  • 認証：bearer
  • 定番：204 No Content（本文なし）
  • ヘッダ：Authorization
  • 別解：DELETE /consents/{id} / 200 + JSON
