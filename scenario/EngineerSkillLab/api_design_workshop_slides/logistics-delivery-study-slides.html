API設計ワークショップ

物流・配送APIをRESTで設計する

アプリ実装準拠：出荷・追跡・イベント記録・取消の5問を整理

シナリオの前提（実装準拠）

  • EC倉庫システムと配送キャリアをつなぐバックエンドAPI
  • 出荷・追跡・イベント記録・取消は 関係者のみ 実行可
  • この課題では 全エンドポイントが bearer 前提

エンドポイント全体図（5本）

一覧取得：GET /shipments

  • 用途：出荷依頼（荷物）の一覧を取得する
  • 認証：bearer
  • 成功：200 OK + JSON
  • ヘッダ：Accept + Authorization

新規登録：POST /shipments

  • 用途：新しい出荷依頼を登録する
  • 認証：bearer
  • 成功：201 Created + JSON
  • ヘッダ：Content-Type + Authorization

追跡取得：GET /shipments/{id}/tracking

  • 用途：追跡番号で配送ステータスを取得する
  • 認証：bearer
  • 成功：200 OK + JSON
  • 別解：GET /tracking/{id}

イベント記録：POST /shipments/{id}/events

  • 用途：配送完了などのイベントを記録する
  • 認証：bearer
  • 成功：201 Created + JSON
  • 別解：POST /deliveries/{id}/complete

出荷取消：DELETE /shipments/{id}

  • 用途：出荷依頼を取り消す（キャンセル）
  • 認証：bearer
  • 定番：204 No Content（本文なし）
  • 別解：200 + JSON

ワークショップ回答例（実装準拠5本）

用途MethodPathAuthStatusHeaders
出荷一覧 GET /shipments bearer 200 Accept + Authorization
出荷登録 POST /shipments bearer 201 Content-Type + Authorization
追跡取得 GET /shipments/{id}/tracking bearer 200 Accept + Authorization
イベント記録 POST /shipments/{id}/events bearer 201 Content-Type + Authorization
取消 DELETE /shipments/{id} bearer 204 Authorization

おつかれさまなのだ！

物流・配送APIは「全 bearer 前提」の切り替えを正確に扱うと、
ヘッダとボディ有無の設計ミスを減らせる。
