導入（incident）

FW は通したが、HTTP の中身の攻撃は見えない——そこで WAF

第17回の通り ファイアウォール（FW）は IP／ポート中心。Web の URL やパラメータに混ざる SQLi・XSS などは アプリ層（L7） で検査する別箱が説明されやすい

水色＝正規の HTTP、オレンジ＝攻撃っぽいペイロード、中央の盾＝WAF の検査イメージ。

  • WAF＝Web Application Firewall（Web アプリケーション向けの検査装置／サービス）。
  • 名称問題では リバースプロキシ型やクラウド WAFなどの設置形態も文献で並ぶ。
第18回（opening）

【登録セキスペ 第18回】WAF の正体 — FW では止まらない攻撃を止める仕組み

Season 3「鉄壁のネットワーク構築編」：第17回で FW、今回はその「向こう側」の HTTP 攻撃を扱う

ゴール： FW と WAF のレイヤの違いを言える。検知の限界（誤検知・未知・ビジネスロジック）を混同しない。安全な実装の代替にならないと説明できる。

見どころ①
OSI 参照モデルで FW と WAF の見える範囲を言語化
見どころ②
誤検知・運用（例外ルールが緩むとリスク）もセットで語られる
定義

WAF は「HTTP を解釈して、攻撃っぽいリクエストをブロック／ログする仕組み」

試験では プロキシとしてクライアントとサーバの間に入る説明や、クラウドエッジで無効化する説明が並列で出る

  • OWASP などで語られる 代表的な Web 攻撃（SQLi、XSS、パストラバーサル等）のパターン検知が説明の定番。
FW との対比

「どこを見ているか」：L3/L4 と L7 を切り分ける

同じ「通信の途中に箱」を置いても、FW が見る情報と WAF が見る情報は別物、という説明が試験向き

UTM／次世代 FW が L7 まで一部見る、という文脈では整理がややこしくなるので、選択肢の語句に合わせて主役の説明を選ぶ。

検知のイメージ

URL・パラメータ・ヘッダ等に現れる 攻撃パターンをルールで照合

第5回 SQLi、第6回 XSS のペイロードが「HTTP のテキスト」として現れる、という接続で覚える

限界と運用

WAF は魔法の盾ではない：未知・誤検知・論理攻撃は別問題になりやすい

試験で頻出なのは「セキュアコーディングの代替ではない」「誤って正当通信を止める」系の語り口

取りこぼし
未知の攻撃、巧妙なエンコーディング、アプリ固有のビジネスロジック欠陥（IDOR 等）はルールだけでは弱い、という説明。
誤検知・運用
例外を増やすと穴が開くジレンマ。ログ監視・チューニングがセット。
暗号化通信
WAF が中身を見るには 終端（TLS）の置き方が必要、という文脈が文献に出る。
多層防御
FW・WAF・入力検証・最小権限・モニタリングを組み合わせる、が王道のまとめ。
試験ひっかけ

短絡しやすい誤答パターン（WAF 編）

× WAF さえ入れればアプリは安全
実装バグ・認可・設定ミスは別経路。根本対策はアプリ側。
× FW と WAF は同じ役割
典型像では 見るレイヤが違う（語句問題で分離）。
△ クラウド WAF ならオリジンは不要
説明抜きで「オリジンを塞ぐ」イメージだけ持つと落ちる。
△ WAF があれば SQLi は100%防げる
バイパス事例・複雑な文脈・ストアド等、説明上絶対視しない。
シリーズ接続

第5・6・17回とつなぐと「多層防御」の説明が通る

  • 根本対策はアプリ（入力処理・認可・セッション等）。WAF は入口でのフィルタ。
