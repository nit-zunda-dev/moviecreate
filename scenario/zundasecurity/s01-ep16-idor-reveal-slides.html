【登録セキスペ 第16回】IDOR — 不適切な直接オブジェクト参照

第15回は XML読み取りが外へ広がる穴。今回は オブジェクトIDを信じすぎて、他人のデータが返る穴。試験では「サーバ側の認可」が押さえどころ。

  • 見どころ：認証≠認可・間接参照・UUIDの限界・第7・12・13・15回との違い
  • 図の見方：ログイン → 要求の id →（持ち主検証なし）→ 他人データ

より詳しい図解: s01-ep16-idor-study-slides.html（同フォルダ）

二段

認証（誰？）と 認可（そのデータを見ていい？）は別

HTTPS やログだけでは 認可の穴は塞がらない。記述の本命は「取得直前の検証」。

直接参照

URL・API・ファイル名に 連番や主キー がそのまま出ると、いじるだけで届く

危ない
/api/orders/10042 を 10043 に変えるだけ、など。
望ましい
返す前に テナント／所有者 と突き合わせる。
権限の軸

同格ユーザー横の越え（水平）／役割の越え（垂直に近いイメージ）

試験では大枠 アクセス制御の不備。用語より「その操作をしていい主体か」を毎回確認できるか。

発生箇所

請求・医療・学務・マルチテナント、モバイルAPI、バッチの署名なしリンク

  • フロントでボタンを隠しただけでは防げない（直接叩かれる）
  • GraphQL でも、リゾルバの根で認可が抜けると同じ
対策①

オブジェクト返却の直前に 認可ゲート（中央ポリシーでもハンドラ内でも）

推奨
「この利用者がこの id を読めるか」を必ず通す
避ける
プロトタイプの「id で取って表示」がそのまま本番
対策②

間接参照・短寿命トークン・セッションに紐づくマップ

  • 外部に出すのは推測困難な参照子、内部は主キーをサーバだけが持つ
  • 第12回（リダイレクト先の許可）とは軸が違う——表を二列で整理
UUID

ランダム id は 推測対策の補助。横流しの URL では突破されうる

試験の「UUID だけで十分」は 疑ってかかる。本命は認可の実装。

対策③

監査ログ・連続アクセスの検知・レート制限 — 止めるのは認可

ログは証拠と改善材料。境界防御（WAF）だけに賭けないのは第15回と同じ考え方。

混同ゼロ

「どの欠陥か」で切る

  • IDOR（今回）：id の空間で持ち主確認が抜ける
  • 第13回：パスを継ぎ足して OS 上の読み取り範囲をずらす
  • 第7回 CSRF：本人ブラウザを騙して依頼を送るのが主
  • 第15回 XXE：XML読み取りが外へ広がる
試験

短絡しやすい誤答パターン

  • ① ログインできていれば安全 — 認可が別問題
  • ② UUID なら完全 — URL 横流しで意味が薄い
  • ③ フロント非表示で十分 — API 直叩き
  • ④ HTTPS＋WAF で完結 — アプリの検証が本体
