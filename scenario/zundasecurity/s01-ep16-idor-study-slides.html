仕組み①

認証（誰？）＋ 認可（そのデータを見ていい？）— の二段

IDOR は多くの場合、後者が抜けている

試験で紛らわしい
HTTPS、ログ、WAF だけ並べる答え — 認可の実装が問われているか読み分ける。
本命のキーワード
返却直前のチェック、テナント／所有者との突合。
欠陥の核

URL・API・ファイル名に、連番や主キーがそのまま出る

利用者が id を変えるだけで、別オブジェクト に届く

  • 例: /api/invoices/10042 → 10043 に変更
  • 添付ファイルが /files/2025/12345.pdf のように推測可能
  • GraphQL でも、リゾルバが「渡された id をそのまま信じる」と同型の穴
権限の軸

同格ユーザーのデータ横断（水平）／役割の越え（垂直に近いイメージ）

試験では大枠 アクセス制御の不備 — ラベルより「毎回サーバで検証」が書けるか

IPA の文脈では Broken Access Control の代表例として IDOR が挙がることが多い。水平・垂直の厳密分類より、その主体がその操作をしてよいか を説明できることが重要。

どこで

マルチテナント SaaS、医療・学務、請求、モバイル API、バッチの一時 URL

フロントで非表示にしただけでは防げない（直接 HTTP を叩かれる）

  • 管理者向けエンドポイントが一般トークンでも応答してしまうパターン（垂直寄り）
  • 共有リンクの「署名なし・期限なし」トークン
対策①

オブジェクト返却の直前に、必ず通る認可ゲート

中央のポリシーエンジンでも、各ハンドラ内でもよい — 抜け道が無いこと

推奨
「この tenant_id / owner_user_id と要求 id は一致するか？」
落とし穴
プロトタイプの「id で取って表示」が本番のまま残る。
対策②

間接参照マップ・短寿命トークン・セッション束ね

外部に出す参照子と内部主キーを分離 — URL 漏洩時に 失効 できる設計も加点されやすい

第12回オープンリダイレクトは「どこへ飛ばすか」の話。今回は「誰のデータか」— 表を二列で整理。

よくある誤解

UUID やランダム id は推測対策の補助に過ぎない

メール・ログ・共有チャットに載った URL は そのまま横流し で突破されうる

試験のひっかけ
「ランダム化すればアクセス制御は不要」系の選択肢 — 誤答になりやすい。
対策③

監査ログ・連続アクセス検知・レート制限

止めるのは認可。ログは検知とフォレンジックスの材料

  • 短時間に連番 id を列挙するアクセスの検知
  • WAF や境界防御だけに賭けない（第15回と同じ姿勢）
混同ゼロ

「どの欠陥か」で切る — 第7・12・13・15回との違い

  • IDOR（今回）：サーバが用意した id 空間 で持ち主確認が抜ける
  • 第13回 パストラバーサル：パス文字列の継ぎ足しで OS 上の読み取り範囲 をずらす
  • 第7回 CSRF：本人ブラウザを騙して 依頼を送らせる のが主眼
  • 第15回 XXE：XML 読み取りが 外ファイル・URL へ広がる
試験

短絡しやすい誤答パターン

  • ① ログインできていれば安全 — 認可が別問題
  • ② UUID なら完全 — URL 横流しで突破
  • ③ フロント非表示で十分 — API 直叩き
  • ④ HTTPS＋WAF でアクセス制御完結 — アプリ側検証が本体
