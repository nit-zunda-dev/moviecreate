第12回

【登録セキスペ 第12回】オープンリダイレクト

第11回 SSRF ＝ サーバが悪いURLを開く。今回 ＝ 302 と Location で、利用者のブラウザが悪い次のページへ飛ばされる。

  • 学習の軸：return / next など、行き先の文字列をそのまま受け取る設計の危険性
  • 図の見方：要求 → サーバの応答 → ブラウザの移動、の矢印
302 と Location

リダイレクト＝サーバの返答の「次のURL」をブラウザが辿る

  • 壊れやすい導線：ログイン後の戻り、OAuth、公式ドメインを一度通すメールのリンク など
3分類

CSRF / SSRF / オープンリダイレクト

4ステップ

拡散 → 本物で認証 → 無検証のLocation → 偽のサイト

悪用①

フィッシングでの使い方

XSS 等との違い

スクリプト奪取（XSS など）と、遷移先の悪用（今回）は別。長文の問題では、どちらの対策かを区別する。

悪用② / OAuth

redirect_uri ＝ リダイレクト信頼境界

用語の枠は違うが、届けてよい先を先に決める、という考え方は同じ。

本物のドメイン

本物のドメインを一度通す

表示は公式のドメイン → 注意が一瞬ゆるみやすい。

対策①

届け先は許可集合（ホワイトリスト）

まず守る
同一オリジン、相対パスだけ、登録したホストだけ、など 許可リストで照合
すり抜け例
//evil、\\、javascript: など。正規化の前後で形が変わる。

禁止だけのリストより、「行ってよい所を先に決める」方が扱いやすい、という文脈が多い。

対策②

意図ID ＋ サーバ内の表

利用者に生のURL を渡さず、番号だけ受け、サーバの表で解決する。

悪い例
return=https://何でも → そのまま Location へ
良い例
相対パスだけ、または id=3 だけ など
試験

3つの取り違え

  • ① CSRF 対策があっても、行き先の確認は別の話
  • ② CORS は「どのサイトが読み取れるか」— 今回はページ遷移
  • ③ SSRF 対策（サーバの外への通信）は別 — 主役はブラウザ

選択肢：リダイレクト先をどう限定するか、の話かを見る
