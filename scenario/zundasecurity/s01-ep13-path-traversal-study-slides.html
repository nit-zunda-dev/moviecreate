導入（incident）

例：「資料ダウンロード」が ../../../etc/passwd を指したら？

見た目は「会社のPDF」のはずが、サーバの設定ファイルが返る

オレンジ枠＝Web公開フォルダ（許された範囲）。赤い札＝本来見せたくないシステム領域。矢印は「パス結合の結果」。

仕組み①

Webルート（公開フォルダ）と、OS上の絶対パス

URLの /images/logo.png は、サーバでは別の長いパスに対応している

  • 相対パス成分 ../ は「一つ上のフォルダへ」。連ねれば公開ルートの外へ出られる。
  • 試験用語: ディレクトリトラバーサル / パストラバーサル（同系の話として出る）。
仕組み②

../ が「階段を降りてから横に出る」イメージ

正規化前にチェックすると抜ける——だから「解決後のパス」で判定する、が定番の型

すり抜け

URLエンコード・二重デコードでフィルタをかわす

%2e%2e%2f は ../ と同じ。検査と解釈の順序がずれると穴になる

初心者メモ
「.. を禁止」だけでは、Unicode・正規化・別デリミタの変種で抜ける事例が試験に載りがち。
どこで起きる？

ファイル取得・テンプレ読込・ログ出力パス … 「文字列をそのまま結合」した所

図: 代表的な3カテゴリ

攻撃4ステップ

① 入力点を見つける → ② ../ を試す → ③ 正規化・エンコードを調整 → ④ 秘密ファイルを読む

「読む」が目的：ソース・設定・鍵・個人情報CSV など

Unix / Windows

区切り文字と「親」の表現が違う——試験は両方の例が出る

/ と \ ・ドライブレター（C:）に注意

現場では「アプリがどのAPIでパスを開いているか」で挙動が変わる。試験では 原理（正規化してから判定） が正解側になりやすい。

対策① 正規化・固定ルート

実パスを解決したうえで、「ベースディレクトリ配下か」を検証する

NG: 入力に「禁止文字」だけ — OK: resolve() 後にプレフィックス一致

やることの型
・公開ルートを1つ決める
・結合 → 正規化（.. 解消）
・結果がルートの子孫パスか確認
・そうでなければ拒否
よくある誤り
正規化前に ../ を文字列検索して安心する、→ エンコードで回避される。
対策② 間接参照

パス文字列を受け取らない——内部IDだけ渡し、サーバが実ファイルを引く

第12回の「意図ID＋表」と考え方が似ている（外部文字列をそのまま使わない）

用語の整理

LFI・RFI・パストラバーサル——試験では「効く場所」で切る

一枚の表で整理（矢印のラベルが命）

任意ファイル読取 （広い親概念として語られることも） LFI（ローカルファイルインクルージョン） include/require がサーバ内のパスを開く PHP等の言語機能とセットで出題されやすい RFI（リモート〜） 外部URLからコードを取り込む設定ミス 第11回 SSRF は「サーバがHTTPで取りに行く」別軸
試験でつまずきやすい所

対策の「文言」に騙されない——禁止リストだけ／正規化なし／Cloudでも発生

正しい記述は「解決後パスでの検証」「間接参照」「最小権限」寄り

① 「WAFがあれば安心」ではない
エンコード変種・アプリ層の結合ミスは別問題。
② SSRF 対策との混同
SSRF＝外向きHTTPの届け先。本回＝ローカルパス開封。キーワードで切る。
③ 第12回オープンリダイレクトとの混同
リダイレクト＝ブラウザのURL遷移。パストラバーサル＝サーバが開くファイルのパス。別カテゴリ。
