【登録セキスペ 第9回｜Season2】セキュリティヘッダを攻略——HSTS・CSP・X-CTO・Referrer-Policyの試験ポイント

HTTP レスポンスヘッダでブラウザの挙動を安全側へ寄せる（動画シナリオと同一タイトル）

本回は Webサーバがレスポンスに付与するセキュリティ関連ヘッダがテーマ。アプリの入力検証や出力エスケープなどアプリケーション対策と別軸で、ユーザエージェント（ブラウザ）のデフォルト挙動を締める役割を持つ。試験では 正式なヘッダ名・目的・ブラウザにどう効くかの三点がセットで問われやすい。

この回の位置づけ
Season2：Webアプリの弱点編・第9回。シリーズは「ずんだもんとめたんのセキスペ」想定。教材YAML: s01-ep09-security-headers.yaml
第8回（直前）とのつながり
クリックジャッキングで学んだ X-Frame-Options や CSP の frame-ancestors も、本回の「レスポンスヘッダ横断」の一部として位置づけ直す。

暗記の仕方（動画内の整理）：名前の羅列ではなく、次の5つの棚に分類して脅威と対応づける。

① Transport（通信経路・HTTPSの維持） ② MIME（Content-Type と実体解釈） ③ 埋め込み（iframe 等の親子関係） ④ 読み込み元制限（CSP の script-src 等） ⑤ Referer・付随情報（Referrer-Policy 等）

加えて文献では Permissions-Policy（旧称 Feature-Policy）によるブラウザ強力機能の許可制御もセキュリティ文脈で説明される。万能対策ではなく多層防御の一層である点も本回の軸。

  • 学習目標①：Strict-Transport-Security（HSTS）が HTTPS への接続を継続しやすくすること、および 初回アクセス前の中間者は別論点であることを説明できる。
  • 学習目標②：X-Content-Type-Options: nosniff が MIME スニッフィング抑制であり、暗号化や認証そのものではないと区別できる。
  • 学習目標③：CSP と X-Frame-Options の役割を混同せず、frame-ancestors と script-src / default-src の論点を分けて語れる。
  • 学習目標④：Referrer-Policy が Referer の送り方を制御し、URL に紛れうる情報の漏えいを減らしうることを理解する。

操作：→ ← Space または画面下のボタン。収録はブラウザ全画面表示が扱いやすい。

正規のHTTPS応答で一度受け取ると、一定期間は HTTP へ降りにくい。ブックマークやリダイレクトの癖とセットで効く。

埋め込み（クリックジャッキング）
X-Frame-Options または CSP frame-ancestors で、親フレームからの表示を制限。

Referrer-Policy と Permissions-Policy

URL に紛れがちな情報の漏えい抑制・ブラウザ機能の許可制御

  • Permissions-Policy（旧 Feature-Policy）：カメラ・位置情報・フルスクリーン等の機能利用をページ単位で制御する説明が文献に出る。

まとめ（summary）

第9回まとめ — セキュリティ関連HTTPレスポンスヘッダ

それぞれ「ブラウザに何を強制・抑制するか」を一文で言えるようにする（試験は言い換え問題が多い）

定義の整理：ここで扱うのは主に HTTP レスポンスに含まれるヘッダで、サーバからユーザエージェント（ブラウザ）へ「より安全な挙動」を指示する。アプリの実装修正だけでは足りない論点を、クライアント側のデフォルトから締める。すべてを防ぐ万能スイッチではない——出力エスケープ・CSRF トークン・認証設計などと役割分担（多層防御）で覚える。
Strict-Transport-Security（HSTS）
効果：一度正規の HTTPS 応答で受け取ると、ブラウザが同一ホスト名への接続を HTTPS 側に寄せやすくする（max-age 必須。必要に応じ includeSubDomains 等）。
限界：初回訪問前にユーザーが騙されて接続した場合など、文脈によっては別途対策が必要。
試験：「HTTPS を強制する」と誤解しやすいが、実際はブラウザの挙動をHTTPS 維持へ誘導するヘッダ。
X-Content-Type-Options: nosniff
効果：Content-Type の宣言を尊重し、MIME スニッフィング（実体を見て種類を再判定する挙動）による解釈のずれを抑える。
典型の誤答：「通信の暗号化」「認証を行う」——別物。
試験：「スニッフィング」「nosniff」の語がセットで出やすい。
Content-Security-Policy（CSP）
効果：script-src / default-src 等で読み込めるリソースの出所を制限。XSS 被害の影響範囲を小さくする補助になりうるが設定・運用は難しいこともある。
第8回との接続：frame-ancestors で親フレームからの埋め込みを制限し、クリックジャッキング対策の本命の一つとして説明される。
X-Frame-Options（XFO）
効果：DENY / SAMEORIGIN 等で、他サイトの frame 内への表示を制限（クリックジャッキング対策として古典的に扱われる）。
試験・実務：文献では CSP の frame-ancestors が現代的な説明として併記されることが多いが、名称と効果は問われうる。CSP 全体と混同しない。
Referrer-Policy
効果：遷移時に送られる Referer（参照元） の量・粒度を制御。別オリジンへURL クエリに載ったトークンや内部パスが流れないよう漏えいリスクを下げる整理ができる。
試験：プライバシー文脈だけでなくセキュリティ（情報漏えい）の観点でも語られる。
Permissions-Policy
効果：カメラ・マイク・位置情報・全画面表示など、強力なブラウザ機能を文書・iframe 単位で許可・拒否する（文献では旧称 Feature-Policy の表記も残る）。
試験：細部のディレクティブより、何を制御するヘッダかの大枠が問われやすい。
  • 脅威対応表（自習用）：第5〜8回の XSS / CSRF / クリックジャッキング と並べ、「主役の対策（エスケープ、トークン、SameSite 等）」と「ヘッダが補助する論点」を1枚に書き出す。
  • ひっかけ再掲：HSTS の初回前限界／CSP と XFO の混同／nosniff は暗号化ではない。
  • 動画クロージング（シナリオ summary）：ブックマークや入口 URL は HTTPS に揃える。ヘッダは複数枚の盾として多層防御の一角。
