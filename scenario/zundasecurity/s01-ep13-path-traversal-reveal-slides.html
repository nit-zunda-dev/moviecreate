第12回は HTTP の次のURL。今回は サーバが開くローカルパス。../ で親に登り、公開ルートの外のファイルを読む。

  • 試験の軸：パス結合・正規化・LFI / RFI / SSRF との切り分け
  • 図の見方：入力 → 結合 → 実パス（resolve 後）
URL と実パス

ユーザー入力がこの変換に混ざると、公開範囲をすり抜けられる。

エンコード

  • 禁止リスト単体より、解決後パスでのホワイトリスト／プレフィックス検証
発生箇所

ダウンロード・include・画像パス — 「ファイル名」のつもりがパス全体に

  • ① ?file= 型の静的配信
  • ② テンプレートのローカル include（LFI 文脈）
  • ③ 添付プレビューなど — ZIPスリップは近親（別語）
攻撃の流れ

入力点 → ../ 試行 → エンコード調整 → 機密ファイル読取

対策①

ベースディレクトリを決める → 結合・正規化 → 子パスか検証

  • 検査が正規化より前だとエンコード変種でズレる — 試験の落とし穴
対策②

パス文字列を渡さない — doc_id=42 だけ受け、サーバが許可表から解決

第12回の「意図ID＋表」と発想は同じ：外部に生の文字列を載せない。

用語整理

パストラバーサル / LFI / RFI / SSRF — 悪い文字列が効く「機構」で切る

  • LFI：include 等でサーバ内ファイルを取り込む
  • RFI：外部URLからコードを取り込む設定ミス
  • SSRF（11回）：サーバがHTTPで届け先を開く — 別軸
試験

混同しやすい三つ

  • ① WAF だけではアプリの結合ロジックは拾えないことがある
  • ② SSRF 対策は外向きHTTPの話 — 今回はローカルパス
  • ③ 第12回オープンリダイレクトはブラウザのURL遷移 — 今回はファイルopen
