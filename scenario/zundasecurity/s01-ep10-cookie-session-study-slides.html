Set-Cookie の属性と「名札」（セッションID）の守り方（動画シナリオと同一テーマ）

第9回までで セキュリティHTTPヘッダがブラウザ挙動を締める話だった。今回は サーバがブラウザに預けるクッキーに焦点を当て、特に セッション識別子がどう盗まれ・便乗されうるかを図で整理する。試験では 属性名より どの脅威の難易度を上げるかが問われやすい。

第9回（直前）とのつながり
HSTS や HTTPS はクッキーの運搬経路を安全側へ寄せる外側の盾。Secure 属性とセットで覚える。

三枚のステッカー（動画内の比喩）：

HttpOnly → JS から読みにくく（XSS窃取の難易度↑） Secure → HTTPS のときだけ送る（平文便乗↓） SameSite → クロスサイト便乗を抑える（CSRF補助）
  • 学習目標①：ステートレスな HTTP に セッションID を載せて状態を引き継ぐ典型像を図で説明できる。
  • 学習目標②：HttpOnly が document.cookie 経路を締める話であり、暗号化そのものではないと言える。
  • 学習目標③：セッション固定化の流れと、認証成功後のセッションID再発行が定番対策だと述べられる。
  • 学習目標④：各属性を 万能対策と誤答しない（多層防御の一片）。

導入（incident）

XSS があると、セッションCookieが「スクリプト目線」で抜けうる

第6回の延長：document.cookie が読める実装は危険

  • ポイント：クッキーは「ブラウザが自動で付ける名札」——奪われればパスワード無しでログイン状態を再現されうる。
  • 第9回との違い：ヘッダはレスポンスのポリシー。クッキーは状態の運搬。
セッション基礎（session_basics）

HTTPはステートレス — セッションIDで「続き」を引き継ぐ

サーバが発行 → クッキーに載せて毎リクエストで返す典型

HttpOnly（httponly）

HttpOnly — 「HTTP(S)の通路」には出るが「JSの窓」は閉じる

document.cookie からの読取・操作を抑止（XSS 窃取対策の一角）

限界：ネットワーク盗聴、XSS以外の経路、ブラウザ拡張などまでは「属性だけ」では保証しない——多層防御。

Secure（secure_flag）

Secure — 「HTTPS の便」にだけ乗せる

平文 HTTP では送りにくくする（経路混線のリスク低減）

SameSite（samesite）

SameSite — 別サイト発のリクエストでの便乗を抑える

第7回 CSRF 対策の「ブラウザ側の補助線」（Strict / Lax / None 等）

セッション固定化（session_fixation）

セッション固定化 — 攻撃者が先に知っているIDを「ログイン後も」使わせる

対策の型：認証成功後にセッションIDを再発行し、旧IDを無効化

多層防御（defense_layers）

属性は「難易度を上げる」— 万能パッチではない

XSS・CSRF・固定化・推測は別論点

  • 試験あるある：「この属性を付ければ XSS が起きない」系は偽が多い。

ひっかけ3点セット

① HttpOnly
暗号化ではない。JS からのアクセス抑制の説明として選ぶ。
② Secure
クッキー値の暗号化ではなく、送信条件（HTTPS）の整理が本命。
③ セッション固定化
対策はログイン成功後のセッションID再発行。他の攻撃の対策と取り違えない。
まとめ（summary）

第10回まとめ — クッキーとセッション管理

「名札」をどう守り、どう取り替えるか

整理：セッションIDは多くの場合 Set-Cookie でブラウザに預けられ、以降のリクエストで自動同梱される。HttpOnly は XSS 経由の document.cookie 読取を抑え、Secure は HTTPS 接続での送信に寄せ、SameSite はクロスサイト文脈での便乗を抑えCSRF対策の補助になりうる。セッション固定化は認証成功後の ID 再発行が試験の定番型。いずれも万能ではない——出力エスケープ・トークン・HTTPS 設計などと役割分担で覚える。
  • 自習：第6〜9回の図と並べ、「ヘッダ」「クッキー属性」「アプリ実装」の三層で脅威を1枚にマッピングする。
  • 動画クロージング：名札は三枚のステッカーで守り、ログインで新しい名札に交換——固定化を潰す。
