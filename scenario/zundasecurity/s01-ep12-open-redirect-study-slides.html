【登録セキスペ 第12回｜Season2】オープンリダイレクト（Open Redirect）を攻略

本番のログイン成功レスポンスが、利用者のブラウザを偽の次のページへ飛ばしてしまう穴

第11回の SSRF は、サーバがURLを取りに行く話。第12回は、HTTPのリダイレクト応答（例: 302 + Location）の行き先の話。図では次の3つ（①誰の要求 ②どのレスポンス ③誰のブラウザが移動）を意識する。

この回の立ち位置
Webの弱点 · 第12回。台本: s01-ep12-open-redirect.yaml
第11回 SSRF との違い（ここ一発）
SSRF ＝ サーバがHTTPで開く。オープンリダイレクト ＝ 利用者のブラウザが、本番の指示で次のURLへ移動する。
  • 学習目標①：302 などと Location を使った、任意の外部URLへのリダイレクトを説明できる。
  • 学習目標②：CSRF・SSRF と、矢印の主役（誰が動くか）を区別できる。
  • 学習目標③：許可リスト・相対パスのみ・IDで解決 など、対策の型を言える。
  • 学習目標④：本物のドメインを通すと信頼されやすい、がフィッシングに使われる理由を図1枚で話せる。

本物のドメイン

図: 正規（緑枠）→ 仕込み（紫帯）→ 被害（赤）

対策① 検証

届け先は許可したものだけ（ホワイトリスト）— 相対URLのみ・正規化に注意

禁止だけのリストより、先に行ってよい所を決める、型が多い

OK の型
・同一オリジンのみ / 登録サフィックス一致
・相対パスだけ受け、絶対URLは捨てる
・パース後のホスト比較（IDN含む文脈）
落とし穴（図で覚える）
//evil、\\、javascript: 等の正規化前後のすり抜け。一発の検査では防げない、という説明が多い。
対策② 設計

意図ID＋サーバ内の表 — URL文字列をLocationにそのまま載せない

図: クライアントは next=3 だけ。実URLはDB。

URLパラメータの心象

悪い例 / 良い例（図1枚で試験の匂い）

外部の文字列そのものを Location に渡さない

悪い例
return=https://（任意）
→ 何でもLocationにコピー
良い例
return=/app/dashboard のみ
又は id=3（サーバで解決）
試験でつまずきやすい所

3つの取り違え — CSRFトークン / CORS / SSRF対策

正解肢は遷移先の妥当性を締める話かどうか

① CSRF 対策 だけ では不十分、がある
トークンは本番操作の贋造向け。リダイレクト先は別問題。
② CORS と混同
CORS＝ブラウザの読取ルール。本回は HTTP リダイレクト遷移の話。
③ SSRF 対策と道具が違う
SSRF＝サーバが外へ出す通信。Open redirect＝利用者のブラウザの移動。出口制御の話をそのまま当てはめない。

第12回まとめ — オープンリダイレクト

Location、本物のドメイン、対策（アプリと設計）の3本

柱① 本番のリダイレクト応答が、利用者のブラウザを悪用先に向ける

柱③ CSRF、SSRF、OAuth の redirect を同じ紙面に一度並べて矢印を引く

1行（口頭チェック）

攻撃：本物でログインしたのち、無検証のLocationで偽先へ。

対策：外部URL文字列をLocationに載せない。許可した先だけ、または意図ID＋表で縛る。
