導入

FW は通したのに Web は攻撃される——HTTP の中身を見るのが WAF

ファイアウォールは IP／ポート中心。WAF（Web Application Firewall）は L7 で URL・パラメータ等を検査し、SQLi／XSS などの典型的パターンをルールで抑える多層防御の一箱。

第18回

WAF の正体 — FW では止まらない攻撃を止める仕組み（Season3）

  • L3/L4 の FW と L7 の WAF を混同しない
  • ルール／シグネチャによる検知と、誤検知・未知攻撃の限界
  • セキュアコーディングの代替ではない（第5・6回と接続）
仕組み

クライアントとオリジンの間で HTTP を解読し、遮断・ログ・レート制限など

設置形態：リバースプロキシ型、クラウド WAF（CDN エッジ）など文献で並ぶ。試験は原理（何を見て何を止めるか）。

検知

代表的 Web 攻撃のペイロードが HTTP に現れる — 第5回 SQLi、第6回 XSS

SQLi っぽさ
クエリパラメータの記号連結などをパターン検知（教材では抽象化）
XSS っぽさ
タグ／イベントハンドラ等の典型的混入を検知
限界

万能ではない — ビジネスロジック欠陥、巧妙なバイパス、誤検知

  • 例外ルールを増やすと緩むジレンマ
  • TLS の終端位置によっては中身が見えない文脈が出る
試験

誤答パターン

  • WAF があればアプリは完全に安全 — ×
  • FW と WAF は同じ — ×（典型像ではレイヤが違う）
  • WAF＝入力検証の代替 — ×（本体はアプリ側）
